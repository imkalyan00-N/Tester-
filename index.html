INTRO TO SCHEMA REFINEMENT & ANOMALIES

Schema Refinement is the process of improving a database schema using well-defined techniques. The most effective technique is decomposition, where a large table is split into smaller, well-structured tables.

Normalization is the process of organizing data by dividing large tables into smaller tables. This ensures the design avoids anomalies, eliminates data redundancy, and maintains integrity.

Redundancy

Redundancy refers to the repetition or duplication of the same data stored in multiple places. Example:

StudentIDStudentNameCourseInstructor
101RaviDBMSRao
102SitaDBMSRao
103KumarDBMSRao

Here, the Instructor name "Rao" is repeated for every student enrolled in DBMS. This is data redundancy.

Anomalies

Anomalies are problems that occur in poorly designed (unnormalized) databases (flat file database).

  • Insertion Anomaly: Occurs when certain data cannot be inserted without inserting other unnecessary data. (Ex: Cannot add a new course unless a student enrolls in it).
  • Update Anomaly: Occurs when updating data in multiple rows leads to inconsistency. (Ex: Changing instructor "Rao" to "Sharma" requires multiple updates).
  • Deletion Anomaly: Occurs when deleting a record unintentionally removes important information. (Ex: Deleting the last student in DBMS deletes the course info too).

Purpose of Normalization:

  • Minimize the redundancy in data.
  • Remove insert, update, and delete anomalies.
  • Reduce the need to organize the data when it is modified.

Advantages:

  1. Greater overall database organization.
  2. Unnecessary redundant data reduced.
  3. Data integrity is easily maintained.
  4. Security is easier to maintain or manage.

Disadvantages:

Produces a lot of small tables requiring primary/foreign key relationships.

  • Performance: Joins slow processing.
  • Complex queries: Developers must code complex queries to merge data.

FUNCTIONAL DEPENDENCY & AXIOMS

Concept of Functional Dependency:

Functional Dependencies are fundamental to the process of Normalization. A functional dependency is a "type of constraint that is a generalization of the notation of the key".

It describes the relationship between attributes (columns) in a table. It is represented by an arrow sign (→).

FD : X → Y

Means that the values of Y are determined by the values of X. Two tuples sharing the same values of X will necessarily have the same values of Y. An attribute on left hand side is known as "Determinant" (Here X is a Determinant).

Armstrong Axioms (Inference Rules):

Armstrong axioms define the set of rules for reasoning about functional dependencies and to infer all the functional dependencies on a relational database.

Primary axioms:

  • 1. Reflexivity: If Y ⊆ X, then X → Y
  • 2. Augmentation: If X → Y, then XZ → YZ
  • 3. Transitivity: If X → Y and Y → Z, then X → Z

Secondary / Derived axioms:

  • 4. Union: If X → Y and X → Z, then X → YZ
  • 5. Decomposition: If X → YZ, then X → Y and X → Z
  • 6. Pseudo-transitivity: If X → Y and WY → Z, then WX → Z

Clear Telugu Explanation

Functional Dependency (FD) ante oka column (X) ni base cheskuni inko column (Y) data thelusukovadam. X ni 'Determinant' antaru. Armstrong axioms anevi FDs ni derive cheyadaniki vaduthamu. (Ex: Transitivity ante X nunchi Y vachi, Y nunchi Z vasthe.. direct ga X nunchi Z vastundi ani artham).

CLOSURE & TYPES OF FDs

Closure of a Set of Attributes (F⁺):

Attribute closure of an attribute set can be defined as the set of attributes which can be functionally determined from it. Written as X⁺.

Types of functional dependencies:

  • 1. Fully Functional Dependency: If the determinant of the FD is either candidate key or super key. (Y cannot determine any subset of X). Ex: ABC → D, but neither A, B, C, AB, BC, AC determines D.
  • 2. Partial Functional Dependency: If a non-prime attribute is getting derived by only a part of the candidate key. (Ex: If key is AB, and B → C, then C is partially dependent).
  • 3. Transitive Functional Dependency: If a non-prime attribute is getting derived by another non-prime attribute. i.e., X → Y, and Y → Z then X → Z holds.
  • 4. Trivial Functional Dependency: Related to Reflexive rule. If Y is subset of X then X → Y holds. Ex: ABC → BC.
  • 5. Multi-Valued Dependency: For each value of X, there is a set of values Y and set of values Z, and Y is independent of Z. Denoted as X ↠ Y / Z.

KEYS IN DBMS

Prime and non-prime attributes: Attributes which are parts of any candidate key of relation are called as prime attribute, others are non-prime attributes.

Candidate Key:

Candidate Key is minimal set of attributes of a relation which can be used to identify a tuple uniquely.

Consider: student(sno, sname, sphone, age)
we can take sno as candidate key.

Types: 1. Simple (one attribute), 2. Composite (multiple attributes).

Super Key:

Super Key is set of attributes of a relation which can be used to identify a tuple uniquely.

  • Adding zero or more attributes to candidate key generates super key.
  • A candidate key is a super key but vice versa is not true.

Consider: student(sno, sname, sphone, age)
we can take sno, (sno, sname) as super key

FINDING CANDIDATE KEYS

Using closure sets to identify keys from Functional Dependencies.

Example 1: Find candidate keys for R(ABCD) having FDs AB → CD, C → A, D → A.

Sol: From the given FD's, the attribute B is key attribute because it is not in RHS of functional dependency. B⁺ = B (not a candidate key, find the combinations of B) AB⁺ = ABCD (∵ AB→CD) BC⁺ = BCAD (∵ C→A, AB→CD) BD⁺ = BDA (∵ D→A) From the above attributes AB and BC determines all attributes. Result: AB, BC are candidate keys.

Example 2: Find candidate keys for R(ABCDE) having FDs A → BC, CD → E, B → D, E → A.

Sol: No attribute is strictly a key attribute because all are in RHS of FD. So check all LHS attributes. A⁺ = ABCDE (A→BC, B→D, CD→E) B⁺ = BD E⁺ = EABCD (E→A, A→BC, B→D) C⁺ = C D⁺ = D CD⁺ = CDEAB (CD→E, E→A, A→BC) BC⁺ = BCDEA (B→D, CD→E, E→A) From the above, A, E, CD and BC determines all attributes. Result: A, E, CD, BC are candidate keys.

Clear Telugu Explanation

Candidate keys ela kanukkovaalo ee examples lo undi. Rule 1: Right Hand Side (RHS) lo leni attributes ni kachitanga key lo undali. Vati closure (+) kanukkunte anni columns vasthe ade key. Raka pothe daniki inko column add chesi malli kanukovali.

NORMAL FORMS (1NF & 2NF)

Points to Remember: 1NF is mandatory, 2NF deals with partial dependencies, 3NF deals with transitive dependencies.

First Normal Form (1NF):

  • Each attribute name must be unique.
  • Each attribute value must be single or atomic (Single Valued).
  • There are no repeating groups.

Second Normal Form (2NF):

A relation is in 2NF if it is already in 1NF and it has no Partial Dependency (i.e., no non-prime attribute is dependent on only a part of the candidate key).

Q1: Given R(ABCD) and F:{AB → C, B → D}. Decompose into 2NF.

Step 1: Find Primary Key. B is not on RHS, so necessary. AB⁺ = ABCD. Primary key = AB.

Step 2: Check 2NF rules. B → D means D (non-prime) is derived by B (part of key AB). This is Partial Dependency.

Step 3: Decompose table. R1(ABC) with key AB, and R2(BD) with key B.

Clear Telugu Explanation

1NF lo okko box lo okate value (Atomic) undali (Ex: Colors box lo "Red, Green" rendu undakudadhu). 2NF loki ravali ante 1NF lo undali, mariyu Partial Dependency undakudadhu. Ante Candidate key lo unna sagam part meedane vere values depend avvakudadhu.

THIRD NORMAL FORM (3NF)

A database is in third normal form if it satisfies the following conditions:

  • It is in 2NF.
  • There is no transitive functional dependency.

By transitive functional dependency, we mean: A is functionally dependent on B, and B is functionally dependent on C. In this case, C is transitively dependent on A via B. (A non-key attribute is depending on a non-key attribute).

Q1: Given R(ABCDE) and F:{AB → C, B → D, D → E}. Decompose into 3NF.

Step 1: Find Primary Key. AB⁺ = ABCDE. Primary key = AB.

Step 2: Check 2NF. B → D is a partial dependency. Decompose to ABC and BDE. Now table is in 2NF.

Step 3: Check 3NF. In BDE, D → E exists. D is non-key, E is non-key. This is Transitive dependency.

Step 4: Decompose to 3NF: R1(ABC), R2(BD), R3(DE).

Clear Telugu Explanation

3NF loki ravali ante table 2NF lo undali, mariyu Transitive Dependency undakudadhu. Ante oka "Non-key attribute inko Non-key attribute ni determine cheyyakudadhu". Ala unte vatini kotha table ga (R3: DE laga) vidagotteyali.

BOYCE-CODD NORMAL FORM (BCNF)

A relation is said to be in BCNF, if and only if every determinant should be a candidate key.

  • BCNF is the advance version of 3NF. It is stricter than 3NF.
  • A table is in 3NF if for every functional dependency X → Y, X is the super key of the table OR Y is a prime attribute.
  • For BCNF, the table should be in 3NF and for every FD (X → Y), LHS (X) must be a super key.

Example Problem:

EMPLOYEE Table: emp_id, emp_nationality, emp_dept, dept_type, dept_no_of_emp FDs: EMP_ID → EMP_NATIONALITY EMP_DEPT → DEPT_TYPE, DEPT_NO_OF_EMP Candidate key: {EMP_ID, EMP_DEPT} The table is not in BCNF because neither EMP_DEPT nor EMP_ID alone are super keys. To convert into BCNF, decompose into three tables: 1. EMP_COUNTRY (emp_id, emp_nationality) 2. EMP_DEPT (emp_dept, dept_type, dept_no_of_emp) 3. EMP_DEPT_MAPPING (emp_id, emp_dept)

Clear Telugu Explanation

BCNF anedi 3NF kante chaala strict rule. Deeni main rule okkate: X -> Y ani unte, X anedi kachitanga Super Key / Candidate key ayyi undali. Leda decomposition cheyyali. Paina example lo EMP_DEPT -> DEPT_TYPE ani undi kani EMP_DEPT single ga key kadu, anduke 3 tables ga vidagottaru.

FOURTH NORMAL FORM (4NF) & SURROGATE KEY

Fourth Normal Form (4NF):

A relation said to be in 4NF if it is in BCNF and should have no multi-valued dependency.

Multi Valued Dependency:

For a dependency A → B, if for a single value of A, multiple values of B exist.

A table should have at-least 3 columns. For R(A, B, C), if there is a MVD between A and B, then B and C should be independent of each other.

Ex: StudentID 21 has courses [Computer, Math] and hobbies [Dancing, Singing]. Course and Hobby are independent, creating unnecessary repetition. Decompose to STUDENT_COURSE and STUDENT_HOBBY tables.

Concept of Surrogate Key:

Alternate of Primary Key that allows duplication of data's/records. It is a unique identification key, like an artificial key to production key (which may be alphanumeric), but the surrogate key is always single numeric key.

  • The value is never reused and is unique within the whole system.
  • It is system generated and an integer.
  • The value cannot be manipulated by the user or application.

Example: Oracle uses SEQUENCE, MYSQL uses Auto_Increment, and SQL Server uses IDENTITY.

Clear Telugu Explanation

4NF loki ravali ante Multi-valued dependency undakudadhu (oka value ki 2 independent lists undadam, like Student ki multiple courses and multiple hobbies unte, rendu veru veru tables cheyyali). Surrogate key ante manam create chesedi kadu, system automatically generate chese unique numbers (ex: MySQL lo Auto_increment).
